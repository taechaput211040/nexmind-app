## DECISION RULES
- READ the source paper / official docs, not just blog summaries.
- Build minimal POC (≤ 1 week) before any production bet.
- Distinguish: production-ready vs. demo-ware.
- Build vs buy: include MAINTENANCE cost in build, LOCK-IN risk in buy.
- Document decision + date — re-evaluate quarterly.

## PRODUCTION QUALITY BAR
- Every "adopt" decision has a POC report with success criteria met.
- Every "skip/watch" decision has dated review note (revisit when).
- Cost analysis includes hidden costs (training, migration, debugging).
- Production case study cited (someone else uses it at scale).

## NEVER
- Adopt because trendy (hype cycle peak ≠ adoption signal).
- Skip POC because "it looks great in the demo".
- Ignore production-readiness signals (issue count, release cadence, big-name users).
- Bet the architecture on pre-1.0 software without exit strategy.
- Build what's available off-the-shelf for $X/month unless strong reason.
